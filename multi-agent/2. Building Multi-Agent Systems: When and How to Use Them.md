## 概述 / Overview

多智能体系统是一种由代码协调、多个大模型实例各自拥有独立对话上下文的架构。协调方式有多种（智能体集群、基于能力的系统、消息总线等），本文聚焦`编排者—子智能体(orchestrator-subagent)`模式：一个主智能体派生并管理多个专门子智能体来处理具体子任务。这种分层模式协调逻辑直观，适合初次接触多智能体系统的团队。

A multi-agent system is an architecture in which several LLM instances each run with their own conversation context, coordinated through code. Several coordination patterns exist (`agent swarms`, `capability-based systems`, `message bus architectures`), but this article focuses on the `orchestrator-subagent` pattern: a lead agent spawns and manages specialized subagents for specific subtasks. This hierarchical model has straightforward coordination logic and is a good entry point for teams new to multi-agent systems.

如今多智能体系统常被用在其实单智能体表现更好的场景中。Anthropic 观察到，有团队花数月搭建复杂的多智能体架构，最后发现只要改进单个智能体的提示词就能达到同样效果。通过实践，作者总结出**三种多智能体确实更优的情形**：**上下文污染拖累性能**时、**任务可并行**时、以及**专门化能改善工具选择或任务聚焦**时。

Multi-agent systems are often applied where a single agent would actually do better. Anthropic has seen teams spend months building elaborate multi-agent architectures only to find that better prompting of one agent achieved the same result. From experience, the authors identify three situations where **multiple agents reliably win**: 
1. when context pollution degrades performance 
2. tasks can run in parallel
3. specialization improves tool selection or task focus

## 优先从单智能体起步 / The Case for Starting with a Single Agent

一个设计良好、配有合适工具的单智能体，能完成的事情往往远超开发者的预期。多智能体系统会引入额外开销：**每多一个智能体，就多一个潜在故障点、多一套需要维护的提示词、多一个意外行为的来源**。

A well-designed single agent with the right tools can accomplish far more than many developers expect. Multi-agent systems add overhead: **every extra agent is another potential failure point, another set of prompts to maintain, and another source of unexpected behavior**.

作者观察到，有团队为规划、执行、审查、迭代分别设立智能体，**结果在每次交接时丢失上下文，花在协调上的 token 比执行还多**。测试显示，完成同等任务时，多智能体方案通常比单智能体多消耗 3 到 10 倍的 token，开销主要来自**跨智能体重复上下文、智能体间的协调消息，以及为交接而做的结果摘要**。

The authors have seen teams build elaborate systems with separate agents for planning, execution, review, and iteration, **only to lose context at each handoff and spend more tokens coordinating than executing**. In testing, multi-agent implementations typically use 3–10x more tokens than single-agent approaches for equivalent tasks, due to **duplicated context, coordination messages, and summarizing results for handoffs**.

## 多智能体系统的决策框架 / A Decision Framework

多智能体架构只有在解决单智能体无法克服的特定约束时才有价值，因此应保留给收益明确、足以抵消额外成本的场景。以下三种模式是作者观察到投资回报持续为正的情形。

Multi-agent architectures add value only when they address specific constraints a single agent cannot overcome, so they should be reserved for cases where the benefits clearly justify the extra cost. The three patterns below are where the authors consistently see a positive return.

### 1. 上下文保护 / Context Protection

大模型的上下文窗口有限，上下文越长，回答质量可能越差。当一个智能体的上下文里堆积了对后续子任务无关的信息时，就发生"上下文污染"。子智能体提供隔离：每个子智能体在自己干净、专注于特定任务的上下文中工作。

LLMs have finite context windows, and quality can degrade as context grows. Context pollution happens when an agent's context fills with information from one subtask that is irrelevant to later ones. Subagents provide isolation: each works in its own clean context focused on its specific task.

文中举例：一个客服智能体在诊断技术问题时还要查询订单历史。若每次查订单都往上下文里塞进数千 token，智能体推理技术问题的能力就会下降。多智能体做法是用一个独立的"订单查询智能体"处理完整订单历史并只提取摘要，主智能体只收到它真正需要的几十到上百 token，上下文保持聚焦。当**子任务产生大量上下文（超过 1000 token）但其中大部分与主任务无关、子任务定义清晰、或属于查询检索类操作时，上下文隔离最有效**。

The example given: a support agent that must look up order history while diagnosing a technical issue. If each lookup adds thousands of tokens, its reasoning degrades. The multi-agent solution uses a separate order-lookup agent that processes the full history and extracts only a summary, so the main agent receives just the 50–100 tokens it actually needs. **Context isolation works best when a subtask generates high context volume (over 1000 tokens) that is mostly irrelevant to the main task, when the subtask is well-defined, or for lookup/retrieval operations**.

### 2. 并行化 / Parallelization

并行运行多个智能体，能探索比单个智能体更大的搜索空间，这对**搜索和研究类任务**尤其有价值。Anthropic 的 Research 功能就采用这种方式：主智能体分析问题并派生多个子智能体并行调查不同侧面，每个子智能体独立搜索并返回提炼后的发现。

Running agents in parallel lets you explore a larger search space than one agent can cover, which is especially valuable for **search and research tasks**. Anthropic's Research feature works this way: a lead agent analyzes a query and spawns subagents to investigate different facets in parallel, each searching independently and returning distilled findings.

但更广的覆盖是有代价的。多智能体系统通常消耗 3 到 10 倍的 token；尽管并行能缩短相对于串行执行的耗时，但**由于总计算量大增，多智能体系统整体往往反而更慢**。并行化的主要好处是**全面性 而非 速度**——当你需要在大信息空间中检索、或从多个角度调查复杂问题时，并行智能体能比受上下文限制的单智能体覆盖更多。代价是更高的 token 消耗和往往更长的总耗时，换来更全面的结果。

But broader coverage has a cost. Multi-agent systems typically consume 3–10x more tokens; while parallelism cuts time relative to running the same work sequentially, the sheer **increase in total computation often makes multi-agent systems slower overall**. The main benefit of parallelization is **thoroughness, not speed**: parallel agents cover more ground than a single context-limited agent, at the price of higher token use and often longer total runtime.

### 3. 专门化 / Specialization

不同任务有时受益于不同的工具集、系统提示词或专业领域。与其给一个智能体配上几十种工具，不如让职责明确、工具集聚焦的专门智能体来提升可靠性。

Different tasks sometimes benefit from different tool sets, system prompts, or **domains of expertise**. Rather than giving one agent dozens of tools, specialized agents with focused toolsets matched to their responsibilities can improve reliability.

- **工具集专门化**：当智能体工具过多（常为 20 个以上）、工具横跨多个不相关领域导致领域混淆、或新增工具反而拖累已有任务表现时，说明需要拆分工具。
- **系统提示词专门化**：不同任务可能需要相互冲突的人设或约束——客服需共情耐心，代码审查需精确挑剔；合规检查需严格遵规，头脑风暴需创意灵活。当单个智能体必须在冲突的行为模式间切换时，拆成各有定制提示词的专门智能体结果更稳定。
- **领域专长专门化**：某些任务需要会让通才智能体不堪重负的深度领域背景（如法律的判例法、医学的临床试验方法），让专门智能体各自承载聚焦的专业知识更合适。

- **Tool set specialization**: split tools when an agent has too many (often 20+), when tools span unrelated domains causing confusion, or when adding tools degrades existing performance.
- **System prompt specialization**: different tasks may need conflicting personas or constraints — support must be empathetic and patient, code review precise and critical; compliance needs rigid rule-following, brainstorming needs creative flexibility. Separating into specialized agents with tailored prompts gives more consistent results.
- **Domain expertise specialization**: some tasks need deep domain context that would overwhelm a generalist (case law for legal analysis, clinical trial methodology for medical research), so specialized agents each carry focused expertise.

文中以多平台集成为例：智能体需跨 CRM、营销自动化和消息平台工作，每个平台有 10–15 个相关 API 接口。一个拥有 40 多种工具的单智能体常常选错、混淆各平台的相似操作；拆成工具集聚焦、提示词定制的专门智能体即可解决选择错误。不过专门化会引入路由复杂度：编排者必须正确分类请求并委派给正确的智能体，路由出错会导致糟糕结果，维护多个专门智能体也增加提示词维护负担。专门化在领域可清晰划分、路由判断明确时效果最佳。

The article uses a multi-platform integration example: agents must work across CRM, marketing automation, and messaging, each with 10–15 API endpoints. A single agent with 40+ tools often picks wrong and confuses similar operations; splitting into specialized agents with focused toolsets and tailored prompts resolves the errors. However, specialization adds routing complexity — the orchestrator must classify requests correctly, misrouting causes poor results, and multiple agents increase prompt-maintenance overhead. It works best when domains are clearly separable and routing is unambiguous.

## 何时已"超出"单智能体架构 / Outgrowing Single-Agent Architectures

除了上述框架，还有一些具体信号表明单智能体模式已被用到极限：
- **逼近上下文上限**：智能体频繁占用大量上下文且性能下降，可能上下文压力就是瓶颈。但近期的上下文管理进展（如压缩 compaction）正在缓解这一限制。
- **管理过多工具**：当工具达到 15–20 个以上，模型会花大量上下文和注意力理解可选项。改用多智能体前，可先考虑工具搜索工具（Tool Search Tool），让 Claude 按需、动态发现工具，可减少高达 85% 的 token 消耗。
- **可并行的子任务**：当任务能自然拆解为独立部分（多来源研究、多组件测试），并行子智能体能带来可观加速。

Beyond the framework, concrete signals suggest single-agent patterns have been outgrown:
- **Approaching context limits**: if an agent routinely uses large amounts of context and performance is degrading, context pressure may be the bottleneck — though recent advances such as compaction are easing this.
- **Managing many tools**: with 15–20+ tools, the model spends significant context and attention on its options. Before going multi-agent, consider the Tool Search Tool, which lets Claude discover tools on demand and can cut token usage by up to 85%.
- **Parallelizable subtasks**: when tasks naturally decompose into independent pieces, parallel subagents can deliver substantial speedups.

这些阈值会随着模型进步而移动，当前的限制是实用指南，而非根本性约束。

These thresholds will shift as models improve; current limits are practical guidelines, not fundamental constraints.

## 以上下文为中心的任务拆解 / Context-Centric Decomposition

采用多智能体架构时，最重要的设计决策是如何在智能体之间划分工作。作者观察到团队常常做错，导致协调开销抵消了多智能体的好处。关键洞见是：拆解工作时应采用"以上下文为中心"的视角，而非"以问题为中心"的视角。

When adopting a multi-agent architecture, the most important design decision is how to divide work between agents. Teams frequently get this wrong, creating coordination overhead that negates the benefits. The key insight: decompose work with a context-centric view, not a problem-centric view.

- **以问题为中心的拆解（常常适得其反）**：按工作类型划分（一个智能体写功能、一个写测试、一个审查代码）会造成持续的协调开销，每次交接都丢失上下文——写测试的智能体不知道某些实现决策的原因，审查者缺少探索与迭代的背景。
- **以上下文为中心的拆解（通常有效）**：按上下文边界划分，处理某个功能的智能体也应处理它的测试，因为它已掌握所需上下文。只有当上下文能被真正隔离时才应拆分工作。

- **Problem-centric decomposition (often counterproductive)**: dividing by type of work (one agent writes features, another writes tests, a third reviews code) creates constant coordination overhead, and each handoff loses context — the test-writer doesn't know why implementation choices were made, the reviewer lacks the exploration context.
- **Context-centric decomposition (usually effective)**: dividing by context boundaries means the agent handling a feature should also handle its tests, since it already has the context. Split work only when context can truly be isolated.

这一原则源于对多智能体失败模式的观察。当智能体按问题类型拆分时，会陷入"传话游戏"，信息来回传递，每次交接都损失保真度。在一次按软件开发角色（规划者、实现者、测试者、审查者）拆分智能体的实验中，子智能体花在协调上的 token 比实际工作还多。

This principle comes from observing failure modes. When agents are split by problem type, they play a "telephone game," passing information back and forth with each handoff degrading fidelity. In one experiment with agents split by software role (planner, implementer, tester, reviewer), the subagents spent more tokens coordinating than on actual work.

- **有效的拆解边界**包括：
    - 独立的研究路径（"亚洲市场趋势"与"欧洲市场趋势"可无共享上下文地并行）；
    - 接口清晰的独立组件（有明确 API 契约时前后端可并行）；
    - 黑盒验证（只需运行测试并报告结果的验证者）。
- **无效的拆解边界**包括：
    - 同一工作的连续阶段（同一功能的规划、实现、测试共享太多上下文）；
    - 紧耦合的组件（需要频繁来回的组件应放在同一智能体）；
    - 需要共享状态的工作（需频繁同步理解的智能体应保持在一起）。

- **Effective boundaries**: 
    - independent research paths ("market trends in Asia" vs. "in Europe" run in parallel with no shared context); 
    - separate components with clean interfaces (a well-defined API contract lets frontend and backend proceed in parallel);
    - blackbox verification (a verifier that only runs tests and reports results needs no implementation context).
- **Problematic boundaries**: 
    - sequential phases of the same work (planning, implementation, and testing of one feature share too much context); 
    - tightly coupled components needing constant back-and-forth; 
    - work requiring shared state.

## 验证子智能体模式 / The Verification Subagent Pattern

一个跨领域都表现出色的多智能体模式是"验证子智能体"——一个专职测试或验证主智能体工作成果的智能体。值得注意的是，能力更强的编排模型（如 Claude Opus 4.5）已越来越能直接评估子智能体的工作而无需单独的验证步骤；但在使用能力较弱的编排者、验证需要专门工具、或希望在工作流中强制设置明确验证检查点时，验证子智能体仍然有价值。

One multi-agent pattern that works well across domains is the verification subagent — an agent whose sole job is testing or validating the main agent's work. More capable orchestrator models (like Claude Opus 4.5) are increasingly able to evaluate subagent work directly without a separate step; but verification subagents remain valuable with less capable orchestrators, when verification needs specialized tools, or when you want explicit verification checkpoints.

验证子智能体之所以成功，是因为它绕开了"传话游戏"问题。验证本质上只需极少的上下文传递，因此验证者可以黑盒测试一个系统，而不需要它如何被构建的完整历史。实现上：主智能体完成一个工作单元后，在继续之前派生一个验证子智能体，交给它待验证的产物、明确的成功标准和执行验证的工具。验证者不需要理解产物为何如此构建，只需判断产物是否满足规定标准。

Verification subagents succeed because they sidestep the telephone-game problem. Verification inherently needs minimal context transfer, so a verifier can blackbox-test a system without the full build history. In implementation: the main agent finishes a unit of work, then spawns a verification subagent with the artifact, clear success criteria, and verification tools. The verifier need not understand why the artifact was built as it was — only whether it meets the criteria.

验证子智能体适用于：质量保证（运行测试套件、代码 lint、按 schema 校验输出）、合规检查（核查文档是否符合政策、按规则检查输出）、输出验证（交付前确认生成内容符合规格）、事实核查（用单独智能体核验生成内容中的论断或引用）。

Verification subagents are effective for quality assurance (running test suites, linting, validating outputs against schemas), compliance checking (verifying documents against policy and outputs against rules), output validation (confirming generated content meets specs before delivery), and factual verification (a separate agent checking claims or citations).

**"过早宣告胜利"问题**是验证子智能体最显著的失败模式：验证者只跑一两个测试、看到通过就宣布成功。缓解策略包括：给出具体标准（"运行完整测试套件并报告所有失败"而非"确保它能用"）；要求全面检查（测试多种场景和边界情况）；加入负面测试（让验证者尝试本应失败的输入并确认其失败）；给出明确指令——"在标记为通过前你必须运行完整测试套件"这类明确要求至关重要，否则验证智能体会走捷径。

The **early-victory problem** is the most significant failure mode: the verifier runs one or two tests, sees them pass, and declares success. Mitigations: concrete criteria ("run the full test suite and report all failures" rather than "make sure it works"); comprehensive checks across multiple scenarios and edge cases; negative tests (have the verifier try inputs that should fail and confirm they do); and explicit instructions — a requirement like "you MUST run the complete test suite before marking as passed" is essential, or verification agents take shortcuts.

## 总结建议 / Moving Forward

多智能体系统很强大，但并非普遍适用。在引入多个协调智能体的复杂度之前，先确认三点：<br>
(1) 确实存在多智能体能解决的真实约束，如上下文上限、并行机会或专门化需求；<br>
(2) 拆解依据的是上下文而非问题类型——按所需上下文而非工作种类来归组工作；<br>
(3) 存在明确的验证点，让子智能体无需完整上下文即可校验工作。<br>

Multi-agent systems are powerful but not universally appropriate. Before adding the complexity of multiple coordinated agents, confirm three things: <br>
(1) genuine constraints exist that multi-agent solves — context limits, parallelization opportunities, or a need for specialization; <br>
(2) decomposition follows context, not problem type — group work by the context it requires, not the kind of work it is; <br>
(3) clear verification points exist where subagents can validate work without full context.

作者的建议是：从可行的最简单方案起步，只在有证据支持时才增加复杂度。

The authors' advice: start with the simplest approach that works, and add complexity only when evidence supports it.
