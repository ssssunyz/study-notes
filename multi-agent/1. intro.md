# Multi-Agent 与 Agent Orchestration 入门

multi-agent 和 orchestration 这两个概念其实建立在一个更基础的东西上——先搞清楚「什么是 agent」,后面就顺了。本文从这里讲起,然后给出各种编排模式的可运行代码。

---

## 第一步:agent 到底是什么

很多人以为 agent 就是「调一次大模型」。不是。普通的 LLM 调用是一次性的:输入 prompt,输出文本,结束。

**Agent = LLM + 工具 + 循环。** 关键在那个「循环」:模型自己决定要调用哪个工具,agent执行后把结果喂回去,模型再决定下一步,反复直到任务完成。步骤不是你写死的,是模型自己规划的。

这是一个最小但完整的 agent 实现: <br>
*注意那个`run_agent()`，后面会经常用到
```python
from anthropic import Anthropic

client = Anthropic()  # 需要环境变量 ANTHROPIC_API_KEY

def run_agent(system_prompt, user_message, tools, tool_functions, model="claude-sonnet-4-6"):
    """一个通用 agent: LLM + 工具 + 循环"""
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.messages.create(
            model=model,
            max_tokens=2000,
            system=system_prompt,
            tools=tools,
            messages=messages,
        )
        messages.append({"role": "assistant", "content": response.content})

        # 模型不再要求调用工具 → 任务结束
        if response.stop_reason != "tool_use":
            text = "".join(b.text for b in response.content if b.type == "text")
            return text

        # 模型要求调用工具 → 执行并把结果喂回去
        tool_results = []
        for block in response.content:  # 有可能有多个tool
            if block.type == "tool_use":  # type可能是 "tool_use"也可能是 "text"
                result = tool_functions[block.name](**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": str(result),
                })
        messages.append({"role": "user", "content": tool_results})
```

`messages` 列表就是这个 agent 的「记忆」; `tools` 是它能用的能力; `while True` 就是那个循环。一个 agent 的全部本质就在这二十几行里。

## 为什么需要 multi-agent

当你想让**一个** agent 干所有事时,会遇到几个真实的问题:

- **Context 会变臃肿变笨**——一个长任务把所有中间结果都堆在一个对话里,模型注意力被稀释,后期容易出错。
- **角色冲突**——一个 system prompt 同时要它「严谨的代码审查员」又要它「有创意的文案」,指令有conflict。
- **工具过载**——给一个 agent 挂 30 个工具,它选错工具的概率显著上升。
- **没法parallel**——single agent 是串行(serial)的,本可以同时做的事只能一件件来。
- **串行容错率为0**——串行(serial)时， 在第40步发生的错误，可能导致前功尽弃。


**Multi-agent 的思路**:把任务拆给多个专门的 agent,每个 agent 有自己独立的 system prompt、自己的工具子集、自己隔离的 context。好处是关注点分离、context 隔离、可并行、每个 agent 可以单独测试。

代价也要知道:协调逻辑变复杂、token 成本上升(多个 agent 各自维护 context)、延迟增加、一个 agent 出错可能污染下游。所以**不要为了用而用**——能一个 agent 解决就别拆。

---

## Orchestration:几种编排模式

「Orchestration」(编排)就是指多个 agent 之间怎么协调、谁调用谁、信息怎么流动。主流有这么几种模式,核心区别在于「协调 agent」和「执行 agent」的角色与信息流动:

| 模式 | 信息流动 | 适用场景 |
| --- | --- | --- |
| 顺序 / 流水线 | A → B → C,上一个输出 = 下一个输入 | 步骤固定的任务 |
| 路由 / 分流 | 路由 agent 判断后交给某个专家 | 先分类、再分流 |
| 并行 / 扇出-聚合 | 同一任务分发给多个 agent,再汇总 | 可并行的子任务 / 多视角 |
| 协调器-工作者 | 协调器动态拆解任务、派给 worker、收集结果 | 事先不知道步骤怎么拆 |

## 模式一:顺序 / 流水线

最简单。前一个 agent 的输出直接当后一个的输入。适合**步骤固定**的任务,比如「写文章 → 审稿 → 润色」。

```python
def sequential_pipeline(topic):
    # Agent 1:写初稿
    draft = run_agent(
        system_prompt="你是内容写作者。根据主题写一篇 300 字的初稿。只输出正文。",
        user_message=f"主题:{topic}",
        tools=[], tool_functions={},
    )

    # Agent 2:挑问题(它的输入是上一步的输出)
    critique = run_agent(
        system_prompt="你是严格的编辑。指出这篇初稿的 3 个具体问题,只列问题。",
        user_message=draft,
        tools=[], tool_functions={},
    )

    # Agent 3:根据问题改稿
    final = run_agent(
        system_prompt="你是改稿专家。根据编辑意见改写文章,输出最终版。",
        user_message=f"原稿:\n{draft}\n\n编辑意见:\n{critique}",
        tools=[], tool_functions={},
    )
    return final
```

注意每个 agent 有**独立的 system prompt 和独立的 context**——写作者不知道编辑的存在,编辑只看到初稿。这就是「关注点分离」。流水线是写死的,模型不参与「下一步是什么」的决策。

## 模式二:路由 / 分流

一个 router agent 先判断请求类型,再交给对应的专家。适合**先分类、再分流**的场景(典型如客服)。

```python
import json

SPECIALISTS = {
    "技术": "你是技术支持专家,解决软件 bug 和报错。",
    "账单": "你是账单专家,处理付款、退款、发票问题。",
    "销售": "你是销售顾问,介绍产品方案和报价。",
}

def routing(user_query):
    # 路由 agent:只做一件事——分类。强制输出 JSON 方便程序解析
    route_raw = run_agent(
        system_prompt=(
            "你是分流员。判断用户请求属于哪一类:技术 / 账单 / 销售。"
            '只输出 JSON,格式 {"category": "技术"},不要任何别的内容。'
        ),
        user_message=user_query,
        tools=[], tool_functions={},
    )
    category = json.loads(route_raw)["category"]

    # 按分类把请求交给对应专家(handoff / 交接)
    answer = run_agent(
        system_prompt=SPECIALISTS[category],
        user_message=user_query,
        tools=[], tool_functions={},
    )
    return category, answer
```

关键点:路由 agent **本身不解决问题**,它只负责「分流决策」。让它强制输出 JSON,是为了让外层 Python 代码能可靠地解析它的决定——这是 multi-agent 里非常常用的技巧:**用结构化输出在 agent 之间传递控制信息**。

## 模式三:并行 / 扇出-聚合

把同一个任务同时发给多个 agent,再用一个 agent 汇总。适合**可以并行的子任务**,或者想要**多视角**的场景(比如同时让「乐观派」「悲观派」「务实派」分析一个决策)。

```python
import concurrent.futures

def parallel_fanout(decision):
    # 三个不同视角的 agent,可以同时跑
    perspectives = {
        "乐观派": "你只分析这个决策的机会和上行空间。",
        "悲观派": "你只分析这个决策的风险和最坏情况。",
        "务实派": "你只分析执行难度和资源成本。",
    }

    def ask(args):
        name, prompt = args
        result = run_agent(prompt, f"决策:{decision}", [], {})
        return name, result

    # 用线程池真正并行调用,延迟 ≈ 单个 agent 而不是三个相加
    with concurrent.futures.ThreadPoolExecutor() as pool:
        results = dict(pool.map(ask, perspectives.items()))

    # 汇总 agent:把三份分析合成一个平衡的结论
    combined = "\n\n".join(f"【{k}】\n{v}" for k, v in results.items())
    summary = run_agent(
        system_prompt="你是决策顾问。综合下面三个视角,给出一个平衡的最终建议。",
        user_message=combined,
        tools=[], tool_functions={},
    )
    return summary
```

并行的价值有两个:**速度**(三个 agent 同时跑,总耗时约等于最慢的一个)和**质量**(强制从多个互相独立的角度看问题,避免单 agent 的思维定式)。

## 模式四:协调器-工作者(最灵活,也最重要)

前三种的步骤都是**你**写死的。Orchestrator-Worker 不一样:一个 orchestrator agent **自己**把任务拆成子任务,决定派几个 worker、各干什么,再收集结果。<br> 
事先不知道要几步、要怎么拆时,就用这个模式。

这是 Anthropic 自己的研究系统采用的架构——orchestrator 负责规划,worker 负责执行。

*注：以下都是合法JSON
```json
JSON 对象:   {"key": "value"}
JSON 数组:   ["a", "b", "c"]      ← 必须是双引号，如果是单引号，json.loads()会报错
JSON 字符串: "hello"
JSON 数字:   42
JSON 布尔:   true
```

```python
import json, concurrent.futures

def orchestrator_worker(research_question):
    # 第一步:orchestrator 把大问题拆成若干子任务(数量由它自己决定)
    plan_raw = run_agent(
        system_prompt=(
            "你是研究协调员。把用户的问题拆成 2-4 个独立的、可以并行调研的子任务。"
            '只输出 JSON 数组,格式 ["子任务1", "子任务2"],不要别的内容。'
        ),
        user_message=research_question,
        tools=[], tool_functions={},
    )
    subtasks = json.loads(plan_raw)

    # 第二步:为每个子任务派一个 worker(各自带工具,这里举例给个搜索工具)
    def do_subtask(task):
        result = run_agent(
            system_prompt="你是调研员。完成分配给你的子任务,给出简洁的事实性结论。",
            user_message=task,
            tools=[SEARCH_TOOL],            # worker 可以有自己的工具集
            tool_functions={"web_search": web_search},
        )
        return task, result

    with concurrent.futures.ThreadPoolExecutor() as pool:
        findings = list(pool.map(do_subtask, subtasks))

    # 第三步:orchestrator 收集所有 worker 的结果,合成最终答案
    bundle = "\n\n".join(f"子任务:{t}\n发现:{r}" for t, r in findings)
    final = run_agent(
        system_prompt="你是研究协调员。把各子任务的发现整合成一篇完整、连贯的报告。",
        user_message=f"原始问题:{research_question}\n\n各项发现:\n{bundle}",
        tools=[], tool_functions={},
    )
    return final

# 上面用到的工具定义示例
SEARCH_TOOL = {
    "name": "web_search",
    "description": "搜索网络获取信息",
    "input_schema": {
        "type": "object",
        "properties": {"query": {"type": "string"}},
        "required": ["query"],
    },
}

def web_search(query):
    return f"[这里接真实搜索 API,返回关于 '{query}' 的结果]"
```

它和「并行」模式的根本区别:并行模式里子任务是**你**写死的三个视角;这里子任务由 orchestrator **运行时动态生成**——同一段代码,问「对比三家云厂商」它可能拆成 3 个子任务,问「分析某公司财务」它可能拆成 5 个。

更进一步,如果 worker 本身也是一个 orchestrator(它再往下拆、再派 sub-worker),就成了**分层(hierarchical)**架构。

## 几个真正决定成败的工程细节

概念之外,实战里这几点最容易踩坑:

- **信息怎么传递。** Agent 之间靠文本/JSON 传递,不共享内存。要么传完整文本(简单但 token 贵),要么让上游 agent 输出结构化 JSON 让下游精准取用。协调类 agent 强烈建议强制 JSON 输出。
- **错误处理。** 一个 worker 失败不该让整个系统崩。每个 `run_agent` 调用都该包 try/except,失败时给降级结果。还要给 `while True` 循环加最大轮次上限,否则 agent 可能陷入工具调用死循环。
- **成本和延迟。** Multi-agent 的 token 消耗是单 agent 的数倍(每个 agent 各维护一份 context)。一个实用技巧是**分级用模型**:orchestrator 用强模型(`claude-opus-4-6`),worker 用快而便宜的模型(`claude-haiku-4-5-20251001`)。`run_agent` 的 `model` 参数就是为这个留的。
- **别过度设计。** 这是最重要的一条。能用单 agent + 多工具解决的,就别拆成 multi-agent。只有当出现真实的角色冲突、context 过载、或确实需要并行时,multi-agent 才值得它带来的复杂度。

## Claude的response的structure
### Anthropic API 响应结构:正常回复 vs 工具调用

理解 `response` 的结构是写对 agent 循环的关键。`response` 是一个 `Message` 对象,整体长这样:

```python
Message(
    id="msg_01...",
    type="message",
    role="assistant",
    model="claude-sonnet-4-6",
    content=[...],            # ← 这就是 response.content,一个 block 列表
    stop_reason="...",        # "end_turn" 或 "tool_use" 等
    stop_sequence=None,
    usage=Usage(input_tokens=..., output_tokens=...),
)
```

关键是 `content` **永远是一个列表**,列表里每个元素是一个 "content block",block 的 `type` 决定它的形状。两种情况的区别就在这个列表里装的是什么。

---

### 情况 A:正常文本回复

模型直接回答,不调工具。`content` 里通常只有一个 text block,`stop_reason` 是 `"end_turn"`。

```python
# response.content —— SDK 对象形式
[
    TextBlock(type="text", text="北京今天晴,气温 25°C。")
]

response.stop_reason   # "end_turn"
```

对应的原始 JSON:

```json
{
  "id": "msg_01ABC...",
  "type": "message",
  "role": "assistant",
  "model": "claude-sonnet-4-6",
  "content": [
    { "type": "text", "text": "北京今天晴,气温 25°C。" }
  ],
  "stop_reason": "end_turn",
  "stop_sequence": null,
  "usage": { "input_tokens": 14, "output_tokens": 20 }
}
```

取文字就是 `response.content[0].text`,或者更稳妥地:
`"".join(b.text for b in response.content if b.type == "text")`。

---

### 情况 B:工具调用

当 Claude 决定用工具时,响应的 `stop_reason` 是 `"tool_use"`,`content` 里会有一个或多个 `tool_use` block。

```python
# response.content —— SDK 对象形式
[
    TextBlock(type="text", text="我帮你查一下北京的天气。"),
    ToolUseBlock(
        type="tool_use",
        id="toolu_01D7FLrfh4GYq7yT1ULFeyMV",
        name="get_weather",
        input={"location": "Beijing"},
    )
]

response.stop_reason   # "tool_use"
```

原始 JSON:

```json
{
  "id": "msg_01XYZ...",
  "type": "message",
  "role": "assistant",
  "model": "claude-sonnet-4-6",
  "content": [
    { "type": "text", "text": "我帮你查一下北京的天气。" },
    {
      "type": "tool_use",
      "id": "toolu_01D7FLrfh4GYq7yT1ULFeyMV",
      "name": "get_weather",
      "input": { "location": "Beijing" }
    }
  ],
  "stop_reason": "tool_use",
  "stop_sequence": null,
  "usage": { "input_tokens": 2156, "output_tokens": 468 }
}
```

`tool_use` block 的三个字段:

- `id` —— Anthropic 为这次工具调用生成的唯一标识,回传结果时要用它对上。
- `name` —— 它想调用的工具名(对应你传进去的某个工具)。
- `input` —— 它根据 `input_schema` 生成的参数对象,你的代码用这些参数去执行工具。

---

### 几个容易踩坑的细节

**1. tool_use 响应里 `content` 不一定只有 tool_use block。**
上面例子里前面还有一个 `TextBlock`——Claude 经常先「说一句」再调工具。也可能完全没有前置文字,`content` 就只有 `[ToolUseBlock(...)]`。所以遍历时必须按 `block.type` 分别处理,不能假设 `content[0]` 就是工具调用。这正是 agent 循环里写
`for block in response.content: if block.type == "tool_use":` 的原因。

**2. 可能有多个 tool_use block。**
如果开了并行工具调用,模型一次响应里可以同时要求调好几个工具,`content` 里就有多个 `ToolUseBlock`。所以代码要用 `for` 循环收集一个 `tool_results` 列表,而不是只处理一个。

**3. 为什么 `messages.append({"role": "assistant", "content": response.content})` 能直接这么写。**
`response.content` 本身就是一个合法的 content blocks 列表,SDK 允许你把这些 block 对象原样塞回 `messages` 当作 assistant 轮次。append 之后:

- 如果 `stop_reason == "tool_use"` → 再 append 一个 `role: "user"` 的消息,里面装 `tool_result` block(用 `toolu_` 那个 id 对上),然后再次调 API。
- 如果 `stop_reason != "tool_use"`(比如 `end_turn`)→ 循环结束,这就是最终答案。

---

### 一句话总结

判断走哪条路看 `stop_reason`,实际要执行什么看 `content` 里的 `tool_use` block。两者要一起用。
