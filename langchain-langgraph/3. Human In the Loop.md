# Human In the Loop

## HITL 中间件
HITL 建立在两个基础设施之上:
- Checkpointer(检查点持久化) —— agent 每一步的状态都会被存盘。这样图执行可以安全地中断,等几分钟甚至几天后再恢复。
- interrupt() 原语 —— 在图执行中抛出一个中断,把控制权交还给调用方,同时把"需要人审核的内容"传出来。

### 四种决定类型
中间件内置了 四种人类响应中断的方式:`approve`(原样批准并执行)、`edit`(修改后执行)、`reject`(拒绝并附带说明)、`respond`(跳过工具执行,把人的回复直接作为工具结果)。
`respond` 比较特殊——它适合那种"工具本身就是占位符,真正的实现是人的回答"的场景,比如一个 `ask_user` 澄清工具。<br>

e.g. 你自己定义的一个**专门用来向人类提问**的工具
agent 在执行任务时,有时会遇到信息不够、需要澄清的情况。比如:

    用户:"帮我订一张去上海的机票"
    agent 心想:哪天的?早上还是晚上的?预算多少?

agent 需要一个机制来"主动问人类一个问题",然后拿到回答再继续。`ask_user` 就是为此而生的。
```python
from langchain_core.tools import tool

@tool
def ask_user(question: str) -> str:
    """当你需要向用户澄清信息时调用此工具。"""
    return ""
```

然后在 HITL 配置里,把它设成 respond:
```python
HumanInTheLoopMiddleware(
    interrupt_on={
        "ask_user": {"allowed_decisions": ["respond"]},
        # ... 其他工具
    },
)
```

### Resume
恢复时用 Command(resume=...) 把人的决定送回去,agent 从断点处继续。整个过程靠 thread_id 把同一个对话关联起来。
<img src="./pics/截屏2026-05-23 下午6.00.33.png" alt="Alt Text" width="650"> <br>

几种decision的 resume 写法:
```python
# approve
{"decisions": [{"type": "approve"}]}

# edit后执行(改参数)
{"decisions": [{
    "type": "edit",
    "edited_action": {"name": "execute_sql", "args": {"query": "新的安全查询"}}
}]}

# reject并给反馈(反馈会进入对话,告诉 agent 为什么不行)
{"decisions": [{"type": "reject", "message": "这个查询会误删数据,应该加 WHERE 条件"}]}

# respond, ask_user 类工具
{"decisions": [{"type": "respond", "message": "11月17号早上"}]}
```
