## 1. NLP 工具与基础任务

### 1.1 spaCy

spaCy 是一个 Python 的开源 NLP 库，定位是"工业级"——追求快、稳、好用，而不是科研论文那种花哨的最新模型。它能做基础文本处理：分词、词性标注、命名实体识别、句法依存分析等等，一行代码就能跑。

和老牌库 NLTK 的区别：NLTK 更偏教学和研究，速度慢、API 比较散；spaCy 是为生产环境写的，一次加载模型之后处理一篇文档可能只要几毫秒。

最小例子：

```python
import spacy

nlp = spacy.load("zh_core_web_sm")   # 加载一个中文小模型
doc = nlp("苹果公司正在考虑以 10 亿美元收购英国一家创业公司。")

for token in doc:
    print(token.text, token.pos_, token.dep_)   # 词、词性、依存关系

for ent in doc.ents:
    print(ent.text, ent.label_)   # 实体文字、实体类型
```

输出大概会告诉你 "苹果公司" 是 ORG，"10 亿美元" 是 MONEY，"英国" 是 GPE。

### 1.2 NER（命名实体识别）

NER（Named Entity Recognition）就是在一段文本里**找出"专有名词"并给它分类**。常见类别：

- PERSON：人名（埃隆·马斯克）
- ORG：组织（OpenAI、清华大学）
- GPE：国家、城市（北京、法国）
- DATE / TIME：时间（2024 年 3 月、明天下午）
- MONEY：金额（10 亿美元）
- PRODUCT：产品名（iPhone）

例子：句子 "Tim Cook 上周访问了北京，宣布将在上海投资 5 亿美元。"

NER 系统会输出：

- Tim Cook → PERSON
- 上周 → DATE
- 北京 → GPE
- 上海 → GPE
- 5 亿美元 → MONEY

**为什么有用**：搜索引擎要知道"苹果"指的是公司还是水果；金融舆情系统要从新闻里抽出"哪家公司、什么时候、多少钱"；客服机器人要从用户的话里抽出订单号、日期、人名。

**实现演进**：规则/词典 → 统计模型（CRF）→ 深度学习（BiLSTM+CRF）→ 预训练模型（BERT 等）。spaCy 内置的就是一个神经网络 NER 模型。

### 1.3 spaCy 模型分发与 HuggingFace

spaCy 模型既可以用官方 CLI 下载，也可以从 HuggingFace 下载——spaCy 官方把所有模型同步发布到了 HuggingFace 上。

**两种安装方式**：

```bash
# 方式 1：spaCy 官方 CLI
python -m spacy download en_core_web_sm

# 方式 2：直接从 HuggingFace 装
pip install "en_core_web_sm @ https://huggingface.co/spacy/en_core_web_sm/resolve/main/en_core_web_sm-any-py3-none-any.whl"
```

两者底层一样，只是来源不同。HuggingFace 上的官方页面是 `huggingface.co/spacy/en_core_web_sm`，下载是个 `.whl` 文件，约 12.8 MB。

**模型命名规则**：以 `en_core_web_sm` 为例：

- `en` = English（语言）
- `core` = 核心管道（含 tokenizer、POS、parser、NER、lemmatizer 等所有基础组件）
- `web` = 训练语料来自网络文本（OntoNotes 等）
- `sm` = small，小号模型（约 13 MB）

其他档位：`md`（中）、`lg`（大）、`trf`（基于 transformer，效果最好但慢）。中文对应的就是 `zh_core_web_sm`、`zh_core_web_md` 等。

**`en_core_web_sm` 的 NER 能力**：

- F1 ≈ 0.846
- 支持 18 类实体（PERSON、ORG、GPE、DATE、MONEY、PERCENT、PRODUCT 等）
- CPU 上每秒能处理几千词

### 1.4 传统 NER vs LLM-based NER 的对比

**传统 NER（如 spaCy）**：专门为 NER 任务训练的小模型（几十 MB 到几百 MB），针对"识别实体"这一件事单独训练。

- 快：CPU 上每秒几千句
- 便宜：本地跑，不用调 API
- 实体类别**固定**：训练时定义了哪些类别就只能识别这些
- 想加新类别（比如"药品名"）要自己标数据重新训练

**LLM-based NER**：用通用大语言模型，通过 prompt 抽取实体。

- 灵活：prompt 里随时定义新实体类别，不用训练
- 理解力强：能处理复杂上下文、歧义、隐含表达
- 慢、贵：每次调用都要走 API，按 token 计费
- 可能"幻觉"：偶尔编造文本里没有的实体

**实务选择**：

| 场景 | 推荐 |
|---|---|
| 高吞吐、固定实体类别 | spaCy 类小模型 |
| 实体类别多变、需要复杂理解、量不大 | LLM |
| 资源极少的小语种 | LLM 通常胜出 |
| 严格不许联网 / 数据敏感 | 本地小模型 |

工业界趋势是**两者结合**：用 LLM 标注高质量数据，再用这些数据训练一个小模型上线，兼顾质量和速度。

## 2. 神经网络训练基础

### 2.1 Loss（损失函数）

模型预测一个东西，跟"正确答案"对比，**两者的差距用一个数值表示，这个数就是 loss**。loss 越小说明预测越准。

**回归例子**：预测明天气温，模型说 28 度，实际 30 度。最常用的"均方误差（MSE）"损失就是 (28-30)² = 4。

**分类例子**：三分类，正确答案是"猫"，模型给的概率是 [猫: 0.7, 狗: 0.2, 鸟: 0.1]，cross-entropy（交叉熵）loss = -log(0.7) ≈ 0.36。如果模型给的是 [猫: 0.99, 狗: 0.005, 鸟: 0.005]，loss ≈ 0.01，明显更小。

**训练神经网络本质就是一件事：调整参数让 loss 变小。**

### 2.2 Gradient（梯度）

参数有几十亿个，怎么知道该往哪儿调？这就是梯度的作用。

梯度是一个**方向**，告诉你"loss 对每个参数的敏感度"。比如某个参数 w 的梯度是 +0.5，意思是 w 增加一点点，loss 会增加 0.5 × 那一点点；那你就该把 w **减小**才能让 loss 降低。这个过程叫**梯度下降（gradient descent）**。

**比喻**：蒙着眼睛站在山坡上，想走到山谷最低点。你只能感受脚下哪边更陡（梯度），然后朝下坡方向迈一小步。重复几百万次就能走到接近最低点的地方。loss 就是海拔，参数就是你的位置坐标。

### 2.3 特征工程（Feature Engineering）

在深度学习火起来之前，机器学习里**最重要的一步**就是特征工程：把原始数据加工成对模型有用的"特征（features）"。

**例子**：预测餐厅明天客流量，原始数据是日期时间戳、天气、是否节假日、过去 30 天客流。直接把"2024-03-15 18:30"扔给模型没用，要造特征：

- 从时间戳拆出：星期几、是否周末、几点钟、是否就餐高峰
- 从天气拆出：温度、是否下雨、空气质量
- 从历史数据造：过去 7 天平均客流、上周同一天客流、最近趋势是涨还是跌
- 节假日 one-hot：春节、国庆、情人节…

**文本特征工程例子**：把一句话变成数值向量——词频、tf-idf、n-gram（连续 N 个词的组合）等。

深度学习的一个革命是"端到端学习"——让模型自己从原始数据学出有用的表示，特征工程的重要性大幅下降。但在**表格数据**（金融、风控、推荐里的结构化字段）上，特征工程依然是核心，比模型选择重要得多。

## 3. Transformer 与 LLM

### 3.1 Transformer 是什么

Transformer 是 2017 年 Google 一篇论文 *"Attention is All You Need"* 提出的**神经网络架构**。现在几乎所有大模型（GPT、Claude、Llama、Gemini、BERT）的基础。

在 Transformer 之前主流是 RNN / LSTM，必须一个词一个词顺序处理，慢且记忆不好。Transformer 的关键创新是**自注意力机制（self-attention）**，让模型可以**并行**处理整句话，每个词都能直接"看到"句子里任何其他词。

**核心组件**：

1. **Tokenization + Embedding**：先把文本切成 token（"我喜欢猫"切成"我"、"喜欢"、"猫"），每个 token 转成一个向量（比如 4096 维的一串数字）。

2. **Position Embedding（位置编码）**：自注意力本身不知道词的顺序，要额外给每个位置加"位置标签"。

3. **Self-Attention（自注意力）**：核心。每个 token 算出 Q、K、V 三个向量，跟其他 token 互相打分，决定"我现在应该多关注谁"。
   - Q（Query）：我现在要找什么信息？
   - K（Key）：我自己是什么样的信息？（用来被别人匹配）
   - V（Value）：如果你匹配上我，你就拿走这个内容。
   - 例子："这只猫累了，它睡着了"。"它"通过自注意力发现与"猫"关联很强，把"猫"的信息融合进来——这样模型知道"它"指代猫。

4. **Multi-Head Attention（多头注意力）**：上面这个过程同时做很多份（比如 32 个"头"），每个头学习不同的关联模式（语法、语义、指代等）。

5. **Feed-Forward Network（前馈层）**：注意力之后接一个普通的两层神经网络，进一步加工每个 token 的表示。

6. **Residual Connection + LayerNorm**：让深层网络训练稳定的技术细节。

7. **堆叠**：把"注意力 + 前馈"作为一层，堆几十上百层。

**三种常见变体**：

- **Encoder-only**（BERT 类）：理解任务，输出表示。适合分类、NER、检索。
- **Decoder-only**（GPT 类）：生成任务，逐个 token 往后写。ChatGPT、Claude、Llama 都是。
- **Encoder-Decoder**（原始 Transformer、T5）：翻译类任务，先编码源语言再生成目标语言。

### 3.2 Layers（层）

神经网络是"一层一层堆起来的"。每一层是一个数学变换，把上一层的输出变成新的表示。每个 transformer 层 = 一个自注意力模块 + 一个前馈神经网络。

参考数字：

- BERT-base：12 层
- GPT-3：96 层
- Llama 3 70B：80 层

层数越多，模型能学到的"抽象层次"越多——前面的层可能在学语法、词形，后面的层在学语义、推理。但层越多越慢、越占显存。

### 3.3 KV Cache（键值缓存）

当模型**生成文本**时是一个 token 一个 token 蹦出来的。生成第 100 个 token 时，它需要"看一眼"前面 99 个 token。理论上要重新算前面 99 个 token 的 K 和 V，但**前面那 99 个 token 没变**，K 和 V 也就不会变。

聪明的做法：算过一次就把每个 token 的 K 和 V 存下来，下次直接拿。这块存储就叫 **KV Cache**。

- 代价：占显存（生成长文本时占得很多）
- 收益：速度大幅提升——长文本生成基本靠它撑着

你跟 ChatGPT 聊天时，每多生成一个字基本就是常数时间，靠的就是 KV cache。

### 3.4 LLM 生成原理与"第一个 token"

LLM 的生成原理一句话：**它本质上是一个"根据前文预测下一个词"的概率机器，反复执行这个动作就生成出整段文字。** 这叫 **autoregressive generation（自回归生成）**。

**步骤**：

1. **预测下一个 token 的概率分布**。给 LLM 输入"今天天气真"，它内部算出一个概率分布：
   - "好"：35%
   - "不错"：20%
   - "热"：15%
   - "冷"：8%
   - ……（词表里每个词都有概率）

2. **采样一个 token**。简单的话选最高概率"好"；实际中会用 temperature、top-p 等采样策略让输出有多样性。

3. **接到输入后面，重复第 1 步**。现在输入变成"今天天气真好"，再预测下一个 token，比如"，"。一直到模型生成"结束"标志或达到长度上限。

**为什么生成时要"看"前面所有 token**：下一个词完全依赖上下文。

- "我打开冰箱，拿出了一瓶___" → "水"、"啤酒"、"牛奶"概率高
- "他走进车库，发动了___" → "汽车"、"引擎"概率高

模型必须看到所有前文才能判断当前最合理的下一个词。自注意力机制让每个新 token "扫一眼"前面所有 token，KV cache 就是把这个过程加速了。

**Response 的"第一个 token"是怎么生成的**：

LLM 看到的输入远不止你打的那句话，它收到的是**完整的、格式化的对话序列**：

1. **System prompt**：告诉模型它是谁、怎么说话
2. **历史对话**：前面所有来回都拼进去
3. **本轮用户消息**：你刚说的话
4. **特殊标记**：告诉模型"现在轮到 assistant 说话了"

简化看长这样：

```
<|system|>
You are a helpful assistant.
<|user|>
今天天气真好
<|assistant|>
[← 模型从这里开始生成第一个 token]
```

模型看到上面整段输入，做一次前向计算，输出一个词表大小的概率分布（10 万 ~ 25 万 token），然后采样得到第一个 token。

**关键推论**：

- prompt 里每个字都在影响第一个 token 的概率分布。同样问"推荐一本书"，前面加"我是医学生"和"我是程序员"，结果完全不同——这就是 prompt engineering 有用的原因。
- 第一个 token 生成后会被接到输入后面，第二个 token 看到的"前文"就包含了刚生成的第一个。如果第一个 token 选偏了（比如"不"开头），后面整段回答都可能跟着偏。

## 4. 硬件与高效微调

### 4.1 显存（VRAM）

**显存**就是**显卡（GPU）上专用的内存**。

- 英文：**VRAM**（Video Random Access Memory）或 **GPU memory**
- 跟电脑内存（RAM）分开，物理上焊在显卡板子上
- 容量比 RAM 小：消费级 8~24 GB（RTX 4090 是 24GB），数据中心卡 40~141 GB（H100 80GB、H200 141GB）
- 带宽极快，是 CPU 内存的几倍到十几倍

**为什么 AI 很关心显存**：模型参数、激活值、梯度、KV cache 全都要放在显存里 GPU 才能算。

- 推理一个 7B 模型（fp16）：参数 ≈ 14 GB
- 推理一个 70B 模型（fp16）：≈ 140 GB，单卡放不下，要多卡
- 训练比推理至少多 3~4 倍显存（梯度、优化器状态）

LoRA、QLoRA、量化这些技术都是为了**省显存**。

### 4.2 LoRA

**背景**：Llama-2-7B 有 70 亿参数，完整微调要更新每个参数，需要 30+ GB 显存，加上优化器状态普通人玩不起。

**LoRA（Low-Rank Adaptation）思想**：**冻结原模型全部参数**，在某些权重矩阵旁边"挂"两个小矩阵 A 和 B。原来的权重是 W，新的等效权重是 `W + BA`。

- W 是 d × d 的大矩阵（比如 4096×4096）
- B 是 d × r，A 是 r × d
- r 取 8、16、32——很小

训练时只更新 A、B，参数量可能只有原模型的 **0.1%~1%**。训练完后还可以把 A、B 合并回 W，推理时不增加任何延迟。

**效果**：很多任务上 LoRA 微调的质量跟全量微调几乎一样，但显存、训练时间、checkpoint 大小都暴降。一个 LoRA 适配器可能只几十 MB，可以为不同任务训练不同的 LoRA，用的时候挂上去就行。

### 4.3 QLoRA

**QLoRA = 量化（Quantization）+ LoRA**。

**量化**：原本每个权重用 16 位浮点（fp16）存，现在用 4 位整数存。精度损失一些，显存占用降到 1/4。

**QLoRA 做的事**：把冻结的大模型权重**量化成 4-bit 存着**（反正不更新它），只有 LoRA 那两小块矩阵 A、B 用高精度训练。

**效果**：

- 原本 65B 模型微调要好几张 A100
- QLoRA 让你在单张 48GB 显卡上微调 65B
- 24GB 卡（4090）能微调 30B
- 是 2023 年开源社区能玩起大模型微调的关键技术
- 性能损失很小

### 4.4 LoRA vs QLoRA 对比表

| 维度 | LoRA | QLoRA |
|---|---|---|
| 核心思想 | 冻结原模型，只训练小的低秩矩阵 A、B | LoRA 基础上把原模型量化成 4-bit |
| 原模型存储精度 | fp16（每参数 2 字节） | NF4（每参数 0.5 字节，4-bit） |
| 显存占用 | 中等 | **极低**（约 LoRA 的 1/3 到 1/4） |
| 训练速度 | 较快 | 较慢（量化/反量化有额外开销） |
| 微调质量 | 接近全量微调 | 接近 LoRA，损失很小 |
| 推理时是否需要原模型 | 是 | 是（量化后的） |
| 典型场景 | 单卡 24GB 微调 7B/13B | 单卡 24GB 微调 33B，单卡 48GB 微调 65B |
| 推荐显卡 | RTX 3090/4090、A100 等 | 消费级卡也能跑大模型 |
| 训练后产物 | 几十 MB 的 adapter 文件 | 几十 MB 的 adapter 文件 |

**直观例子**：微调 Llama-2-70B：

- 全量微调：8 张 A100 80GB，成本极高
- LoRA：2 张 A100 80GB 左右
- QLoRA：1 张 A100 48GB 或 80GB 就够

**怎么选**：

- 显存够 → LoRA，训练快质量稳
- 显存紧张 → QLoRA，能跑就是胜利
- 需要极致质量 → 全量微调

实务中 QLoRA 已经成了"穷人微调大模型"的事实标准。

## 5. 文本检索与评估

### 5.1 TF-IDF 基础

TF-IDF 是一种**给文档里的词打分**的经典方法，衡量"这个词对这篇文档有多重要"。公式：

**TF-IDF = TF × IDF**

- **TF（词频）**：一个词在当前文档里出现了多少次。出现越多，越可能重要。
- **IDF（逆文档频率）**：log(文档总数 / 包含这个词的文档数)。一个词如果在所有文档都出现（"的"、"是"、"the"、"a"），IDF 就很小，没区分度；如果只在少数文档出现（"量子纠缠"），IDF 就大。

**例子**。假设你有 1000 篇新闻：

- 词"的"：在某篇出现 50 次（TF=50），但 1000 篇都有它，IDF = log(1000/1000) = 0。TF-IDF = 0，没用。
- 词"区块链"：在某篇出现 10 次（TF=10），1000 篇里只有 20 篇提到它，IDF = log(1000/20) ≈ 3.9。TF-IDF ≈ 39，很有信息量。

**用途**：

- 搜索引擎排序（Lucene、Elasticsearch 早期核心算法之一）
- 关键词提取（每篇文档 TF-IDF 最高的几个词就是关键词）
- 把文档表示成向量做分类、聚类

现在很多场景被 embedding（向量嵌入）取代，但 TF-IDF 因为简单、可解释、不需要训练，至今仍在很多生产系统使用。

### 5.2 当所有文档都包含某词时 TF-IDF 会怎样

**TF-IDF 会接近 0，几乎没价值。**

如果 1000 篇里 1000 篇都有"区块链"：

IDF = log(1000/1000) = log(1) = 0，TF-IDF = TF × 0 = **0**。

**直觉理解**：如果一个词到处都是，那它就没有区分度——你不能用它来区分这 1000 篇文档里哪篇更相关。就像在一个全是程序员的群里说"会写代码"没意义。

这就是 IDF 设计的核心目的：**惩罚到处都出现的"水词"**。中文里的"的、是、了"，英文里的"the、a、is"，IDF 都接近 0，自动被压成不重要。

**实践注意**：真实场景用 IDF 通常会做平滑，避免 log(1)=0 太极端，常见的是 `log((N+1)/(df+1)) + 1`，这样即使一个词在所有文档都出现也不会完全归零，只是非常小。

### 5.3 RAG chunking 里的 TF-IDF / BM25

在 RAG（Retrieval-Augmented Generation，检索增强生成）里，"文档"概念变成了 **chunk（文本块）**——把一本书切成几百到几千字一段的小块，然后用 TF-IDF 类方法检索。

**变化 1：文档 → chunk**。公式不变，N 现在是 chunk 总数（可能几百万），df 是包含某词的 chunk 数。

**变化 2：实际上很少用纯 TF-IDF，而是用 BM25**。公式（简化）：

```
BM25(q, d) = Σ IDF(t) × [tf(t,d) × (k₁+1)] / [tf(t,d) + k₁ × (1 - b + b × |d|/avgdl)]
```

两个关键改进：

- **TF 饱和**：一个词在 chunk 里出现 50 次不会比出现 10 次重要 5 倍，效益递减（k₁ 控制）
- **长度归一化**：长 chunk 里出现 5 次某词，不如短 chunk 里出现 5 次重要（b 控制，avgdl 是平均长度）

为什么需要？RAG 里 chunk 长度不一样，2000 字的长块自然容易包含查询词，纯 TF-IDF 会偏向长块，BM25 做了归一化。

**变化 3：常和向量检索混合（Hybrid Search）**。纯 BM25 是关键词匹配，"治感冒"匹配不上"治疗感冒"。所以现代 RAG 通常：

- BM25 负责精确关键词召回（专业术语、人名、代号）
- Embedding（向量）检索负责语义召回（理解同义、释义）
- 两者结果合并，再用 **reranker** 重排

实务结论：**BM25 + Embedding 混合检索 > 单独用任何一种**。

### 5.4 ANN 搜索与 HNSW 算法

**ANN = Approximate Nearest Neighbor（近似最近邻搜索）**

**问题背景**：retrieval 的本质是"在几百万个文档向量里，找出和 query 向量最接近的 top-k 个"。

- **精确做法（exact NN）**：query 和每一个向量都算一次距离，排序取 top-k。几百万向量 × 每次 query = 太慢，O(N×D)。

- **ANN**：牺牲一点点精度（可能漏掉真正第 5 近的，给你第 6 近的），换取几个数量级的速度提升。实际应用里这点精度损失基本无感，但速度从几百 ms 降到几 ms。

<br>

**HNSW = Hierarchical Navigable Small World**

HNSW 是目前最主流的 ANN 算法（Pinecone、Weaviate、Milvus、pgvector 都用它）。

**直觉解释**：想象一个**多层的图**结构：

- 最上层节点很稀疏，节点之间是"高速公路"（长距离连接）
- 越往下层节点越密，是"小路"（短距离连接）

搜索时：从最上层入口进，先走高速公路快速接近目标区域，然后逐层下降，在越来越细的层里精确定位。类比就像找地址：先看国家 → 省 → 市 → 街道，而不是挨家挨户问。

**为什么快**：每次跳跃都大幅缩小搜索范围，复杂度从 O(N) 降到约 O(log N)。

**关键参数**：

| 参数 | 含义 | 影响 |
|---|---|---|
| M | 每个节点的连接数 | 越大，精度越高，但内存越大 |
| ef_search | 搜索时探索的候选数 | 越大，越准，但越慢 |

**实务应用**：Pinecone、Weaviate 这样的向量数据库底层就是 HNSW，让你不用关心底层细节，只需指定 k 就能秒级检索百万量级向量。

### 5.5 Precision@k 和 Recall@k

这两个是评估**排序/检索/推荐**系统的指标。完整的 precision 和 recall 加 @k 是只看"前 k 个结果"。

- **Precision@k**：前 k 个结果里有多少是相关的，除以 k
- **Recall@k**：前 k 个结果里有多少是相关的，除以**所有相关结果的总数**

**例子**。搜索"最好的意大利面食谱"。数据库里总共 **20 篇**真正相关的食谱（ground truth）。搜索引擎返回 10 条结果，按相关性排序，其中位置 1、2、4、5、8、10 是相关的（共 6 条）。

- Precision@5 = 前 5 条里有 4 条相关（位置 1、2、4、5）→ 4/5 = **0.8**
- Recall@5 = 4/20 = **0.2**
- Precision@10 = 6/10 = **0.6**
- Recall@10 = 6/20 = **0.3**

**怎么理解差异**：

- Precision@k 关心"我给用户看的前几条质量怎么样"——用户体验
- Recall@k 关心"我有没有漏掉相关内容"——召回能力

通常 k 越大，Precision 下降（越往后越不相关），Recall 上升（捞到的相关结果越多）。这就是 **precision-recall trade-off**。

**怎么选 k**：看产品。Google 搜索很多人只看第一页，Precision@10 关键；推荐系统首屏 5 个位置，Precision@5 关键；学术文献综述更看重 Recall（不能漏文献）。

### 5.6 RAG 中的 Precision@k / Recall@k

在 RAG 评估里，这两个指标是**评估"检索阶段"质量**的核心。

Precision 问"**top k里，有多少是对的**"，Recall 问"**所有对的东西里，我retrieve到了多少**"。

**典型例子**：用户问"请总结《三体》中所有提到'黑暗森林法则'的段落"。整个知识库（《三体》全文切成 5000 个 chunk）里，**真正讨论黑暗森林法则的 chunk 有 10 个**。

你的检索系统返回 top-5：

| 排名 | chunk 内容（简化） | 是否相关 |
|---|---|---|
| 1 | "...宇宙就是一座黑暗森林，每个文明..." | ✅ |
| 2 | "罗辑在雪地里想着..." | ❌ |
| 3 | "...森林法则的核心是猜疑链..." | ✅ |
| 4 | "叶文洁的童年..." | ❌ |
| 5 | "...一旦暴露就会被消灭..." | ✅ |

- **Precision@5** = 3/5 = **0.6**
- **Recall@5** = 3/10 = **0.3**

Recall = 0.3 意味着**还有 7 个相关 chunk 没找到**——LLM 看不到这 7 段，回答可能不完整。

**分母的关键理解**：

| 指标 | 分子 | 分母 |
|---|---|---|
| Precision@k | top-k 里的相关数 | **k**（你返回的数量） |
| Recall@k | top-k 里的相关数 | **总相关数**（知识库里所有相关的） |

**Recall 在 RAG 里的关心点**：

| 指标 | 关心什么 | 提高它的代价 |
|---|---|---|
| Precision@k 高 | LLM 输入干净，不被无关内容带偏 | 可能漏召回信息 |
| Recall@k 高 | 不漏关键信息，答案完整 | 可能塞入噪声，LLM 被干扰 |

**典型架构**：

- **Recall 优先 + Precision 优先两阶段**：先用 BM25+Embedding 召回 top-50（追求高 Recall@50），再用 reranker 精排出 top-5（追求高 Precision@5），最后给 LLM。
- 这种"召回 + 重排"在工业 RAG 里几乎是标配。

**Recall 的评估难点**：分母（总相关数）通常**没法精确知道**——知识库 100 万 chunk，不可能逐个人工标注。所以工业评测会构造**评测集**：人工挑出一批查询，对每个查询人工标注哪些 chunk 相关，在这个小评测集上算 Recall@k。

**常见误区**：很多人只盯着 LLM 最终输出对不对，忽略检索环节的 P@k / R@k。其实 **Recall@k 低是 RAG 失败的最常见原因**——根本没找到相关信息，再强的 LLM 也只能瞎编。做 RAG 第一步就该测检索的 P@k 和 R@k。
